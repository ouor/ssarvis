# PLAN

## 1. 문서 목적

이 문서는 `PRD.md`에 정의된 제품 방향을 실제 구현 순서로 전환하기 위한 실행 계획 문서다.  
핵심 목표는 큰 전환을 한 번에 밀어붙이지 않고, 각 단계마다 작더라도 실제 결과물을 만들고 검증하며 다음 단계로 넘어갈 수 있도록 애자일한 작업 순서를 제공하는 것이다.

## 2. 계획 수립 원칙

- 각 단계는 독립적으로 데모 가능한 결과물을 만들어야 한다.
- 각 단계는 백엔드, 프론트엔드, 테스트 관점의 완료 기준이 있어야 한다.
- 기존 코드베이스의 재사용 가치를 살리되, 새 제품 중심축과 맞지 않는 구조는 억지로 끌고 가지 않는다.
- 제품의 핵심 전환 순서는 `SNS 구조 확립 -> 사람 간 DM -> 사용자 대리 AI 자동응답 -> 음성/부가 기능`이다.
- 자동응답은 텍스트 기반 완성 후 음성으로 확장한다.

## 3. 전체 구현 전략 요약

권장 구현 순서는 다음과 같다.

1. 기반 정리
2. 계정/관계 모델 전환
3. 게시물 도메인 추가
4. DM 도메인 신설
5. 클론/보이스를 사용자 대리 자산으로 재정의
6. DM 자동응답 1차: 텍스트
7. DM 자동응답 2차: 숨김 및 음성 재생
8. 음성 메시지형 대화 및 논쟁 재배치

이 순서의 핵심 이유는 다음과 같다.

- 2~4단계만 끝나도 SNS와 DM의 기본 골격이 눈에 보인다.
- 5~6단계에서 SSARVIS만의 핵심 차별점인 사용자 대리 AI가 붙는다.
- 7~8단계에서 음성과 부가 기능을 더해 제품 완성도를 올릴 수 있다.

추가 구현 원칙:

- 각 단계는 가능하면 `배포 가능한 얇은 슬라이스`로 끝내야 한다.
- 대규모 전환 구간에서는 새 기능을 기존 화면에 억지로 덧붙이지 말고, 새 앱 셸과 새 도메인을 병행 구축한 뒤 전환한다.
- 위험도가 큰 기능은 기능 플래그 또는 내부 전용 노출로 먼저 검증한다.
- 단계 완료 시점마다 `데모 가능 여부`, `자동화 테스트`, `정책 준수 여부`를 함께 확인한다.

## 4. 단계별 구현 계획

### 단계 1. 기반 정리

목표:

- 기존 `클론 체험 서비스` 중심 구조에서 `SNS + DM + 사용자 대리 AI` 중심 구조로 전환하기 위한 코드/용어/테스트 기준을 정리하고, 새 앱 셸의 최소 골격을 만든다.

주요 작업:

- `PRD.md` 기준으로 구현 체크리스트 작성
- 기존 도메인 중 재사용 대상과 폐기/축소 대상을 구분
- `friend` 중심 표현을 장기적으로 `follow` 중심으로 전환하기 위한 용어 매핑 정리
- 기존 프론트 구조 중 `CloneStudioPage` 중심 설계를 어디까지 유지할지 판단
- 기존 테스트를 `유지`, `수정`, `폐기`로 분류
- `Home / Search / DM / Profile / Settings` 기반의 새 앱 셸 뼈대 추가
- 기존 스튜디오 화면과 새 앱 셸을 병행 운용할 수 있는 진입 전략 정리

예상 결과물:

- 구현 체크리스트
- 도메인/용어 매핑표
- 마이그레이션 메모
- 테스트 분류 메모
- 새 앱 셸 스캐폴드 또는 플레이스홀더 화면

완료 기준:

- 팀이 어떤 구조를 유지하고 무엇을 새로 만들지 공통 이해를 가진다.
- 이후 단계에서 `기존 코드 재사용` 여부로 혼선이 생기지 않는다.
- 새 정보 구조를 프론트에서 실제로 탐색할 수 있다.

테스트/평가:

- 문서 리뷰
- 기존 코드베이스 구조 점검
- 테스트 자산 분류 확인
- 새 앱 셸 수동 확인

### 단계 2. 계정/관계 모델 전환

목표:

- 친구 기반 구조를 SNS형 팔로우 구조로 전환하고, 공개/비공개 계정 정책을 구현한다.

주요 작업:

- `follow` 도메인 설계 및 구현
- 사용자 프로필 공개성 상태 추가
- 공개 계정/비공개 계정 접근 제어 정책 추가
- 비공개 계정의 신규 팔로우 불가 정책 반영
- 공개/비공개에 따른 프로필 및 접근 가능성 판단 API 초안 구현

예상 결과물:

- 팔로우 생성/해제 기능
- 공개/비공개 계정 설정 저장
- 계정 공개성 기반 권한 판정 로직

완료 기준:

- 공개 계정은 팔로우 가능하고 비공개 계정은 신규 팔로우가 불가능하다.
- 접근 정책이 서비스 레벨에서 일관되게 적용된다.

테스트/평가:

- 공개 계정 팔로우 성공 테스트
- 비공개 계정 신규 팔로우 차단 테스트
- 공개/비공개 계정 권한 판정 테스트
- 최소 프로필 조회 흐름 수동 확인

### 단계 3. 게시물 도메인 추가

목표:

- SNS처럼 보이는 첫 번째 사용자 가시 결과물인 게시물 기능을 추가한다.

주요 작업:

- 게시물 엔티티/리포지토리/API 추가
- 게시물 작성 기능 추가
- 공개 계정/비공개 계정 기준 게시물 조회 권한 반영
- 프로필 기반 게시물 목록 조회
- 초기 피드 또는 프로필 게시물 UI 구성

예상 결과물:

- 게시물 업로드 가능
- 사용자 프로필에서 게시물 조회 가능
- 접근 권한에 따라 게시물 조회 제한 가능

완료 기준:

- 공개 계정 게시물은 누구나 볼 수 있다.
- 비공개 계정 게시물은 기존 팔로워만 볼 수 있다.
- 최소한 프로필 또는 피드 화면에서 게시물이 렌더링된다.

테스트/평가:

- 게시물 작성 API 테스트
- 게시물 목록 조회 테스트
- 공개/비공개 접근 테스트
- 기본 게시물 UI 확인

### 단계 4. DM 도메인 신설

목표:

- 기존 클론 채팅 도메인과 분리된 SNS용 사람 간 DM 구조를 만든다.

주요 작업:

- `DMThread`, `DMMessage` 성격의 새 도메인 설계 및 구현
- DM 목록 API
- DM 상세 API
- 텍스트 메시지 전송 API
- 공개 계정/비공개 계정 DM 허용 정책 반영
- 최소 DM 목록/상세 UI 추가

예상 결과물:

- 사람과 사람이 DM 가능
- DM 목록 및 대화 상세 확인 가능

완료 기준:

- 공개 계정은 누구나 DM 시작 가능하다.
- 비공개 계정은 기존 팔로워만 DM 시작 가능하다.
- 사람 메시지가 정상 저장되고 재조회된다.

테스트/평가:

- DM 생성 테스트
- DM 전송/조회 테스트
- 공개/비공개 DM 권한 테스트
- 프론트 DM 목록/상세 수동 시연

### 단계 5. 클론/보이스를 사용자 대리 자산으로 재정의

목표:

- 기존 독립 캐릭터 성격의 클론/보이스를 사용자 프로필 소속 자산으로 전환한다.

주요 작업:

- 사용자당 클론 1개 제약 추가
- 사용자당 보이스 1개 제약 추가
- 기존 클론 생성 흐름을 프로필 설정 흐름으로 이동
- 기존 공개 클론/공개 보이스 탐색 개념 축소 또는 제거
- 설문 기반 클론 생성/수정 UX를 프로필 설정에 맞게 재배치

예상 결과물:

- 사용자별 클론 1개 관리 가능
- 사용자별 보이스 1개 관리 가능
- 프로필 설정에서 클론/보이스 관리 가능

완료 기준:

- 동일 사용자가 여러 클론/보이스를 동시에 생성할 수 없다.
- 클론이 독립 캐릭터가 아니라 사용자 설정 자산으로만 다뤄진다.

테스트/평가:

- 단일 클론 제약 테스트
- 단일 보이스 제약 테스트
- 프로필 설정 흐름 확인

### 단계 6. DM 자동응답 1차: 텍스트

목표:

- 텍스트 DM에서 사용자 대리 AI가 실제로 자동응답하도록 구현한다.

주요 작업:

- 사용자 자동응답 설정값 저장
- 마지막 활동 시각 저장 및 갱신
- 부재중 판정 로직 구현
- 사람 메시지 수신 시 자동응답 조건 평가
- AI 메시지 생성 및 저장
- AI 메시지 메타데이터와 `AI` 표시 지원
- AI가 AI 메시지에 응답하지 않도록 차단

예상 결과물:

- 설정에 따라 상대 클론이 DM에 자동응답
- 대화창에서 사람 메시지와 AI 메시지를 구분 가능

완료 기준:

- `항상 / 부재중 / 사용 안함` 설정이 모두 동작한다.
- 부재중 기준은 마지막 API 요청 후 3분으로 동작한다.
- AI는 AI 메시지에 자동응답하지 않는다.

테스트/평가:

- 설정별 자동응답 동작 테스트
- 마지막 활동 시각 갱신 테스트
- 부재중 판정 테스트
- AI 메시지 메타데이터 및 표기 테스트
- 실제 DM 데모 평가

### 단계 7. DM 자동응답 2차: 숨김 및 음성 재생

목표:

- 자동응답 메시지 소비 UX를 다듬고 텍스트 DM의 음성 경험을 추가한다.

주요 작업:

- `유발 메시지 + AI 응답` 묶음 단위 숨김 구현
- 사용자별 숨김 상태 저장 구조 추가
- AI 숨김 UI 추가
- 텍스트 메시지 음성 재생 기능 추가
- 텍스트 기반 AI 응답에 대한 TTS 재생 연결

예상 결과물:

- 사용자가 자신의 화면에서 AI 응답 묶음을 숨길 수 있음
- 텍스트 DM을 음성으로 들을 수 있음

완료 기준:

- AI 숨김은 개인 UI에만 적용된다.
- 유발 메시지와 AI 응답이 함께 숨겨진다.
- 텍스트 메시지의 음성 재생이 자연스럽게 동작한다.

테스트/평가:

- 숨김 단위 테스트
- 사용자별 숨김 독립성 테스트
- TTS 재생 테스트
- 실제 대화 UI 사용성 평가

### 단계 8. 음성 메시지형 대화 및 논쟁 재배치

목표:

- 음성 메시지형 대화를 추가하고 논쟁 기능을 메인 구조 밖의 부가 기능으로 재배치한다.

주요 작업:

- 음성 메시지 전송/조회/재생 기능 추가
- 음성 대화 모드에서 AI 음성 자동응답 구현
- 프로필에서 논쟁 기능에 접근하도록 UI 재배치
- 논쟁 기능을 메인 탐색 흐름에서 분리

예상 결과물:

- 음성 메시지형 DM 가능
- AI 음성 자동응답 가능
- 논쟁 기능이 프로필 기반 부가 기능으로 이동

완료 기준:

- 음성 대화는 실시간 통화가 아니라 음성 메시지형으로 동작한다.
- AI가 음성 대화 모드에서 음성으로 응답한다.
- 논쟁 기능이 메인 SNS/DM 흐름을 방해하지 않는다.

테스트/평가:

- 음성 메시지 업로드/재생 테스트
- AI 음성 응답 테스트
- 논쟁 진입 경로 및 UX 수동 확인

## 5. 스프린트 단위 권장 분할

권장 스프린트 구성은 다음과 같다.

1. 스프린트 1: 기반 정리 + 관계 모델 전환
2. 스프린트 2: 게시물 도메인
3. 스프린트 3: 사람 간 DM
4. 스프린트 4: 클론/보이스 재정의
5. 스프린트 5: 텍스트 자동응답
6. 스프린트 6: AI 숨김 + 텍스트 음성 재생
7. 스프린트 7: 음성 메시지형 대화 + 논쟁 재배치

각 스프린트는 다음 세 가지 산출물을 반드시 가져야 한다.

- 동작하는 기능 결과물
- 자동화 테스트 또는 수동 검증 시나리오
- 데모 및 평가 메모

권장 운영 방식:

- 스프린트 초반에는 도메인/API와 화면 스캐폴드를 먼저 열고, 후반에 정책/테스트를 닫는다.
- 각 스프린트 말에는 다음 스프린트 착수 조건을 명시한다.
- 미완성 기능은 메인 경로에 노출하지 않거나 기능 플래그 뒤에 둔다.

## 6. 단계별 결과물 관점 체크포인트

각 단계 종료 시 다음 질문에 답할 수 있어야 한다.

- 실제로 눈에 보이는 제품 변화가 있는가
- 권한/정책이 코드로 고정되었는가
- 다음 단계가 현재 결과물 위에서 자연스럽게 이어지는가
- 기존 기능과 충돌 없이 마이그레이션 가능한가

단계별 체크포인트:

- 단계 2 종료: SNS 관계 정책이 실제로 작동하는가
- 단계 3 종료: 게시물이 생겨 SNS처럼 보이기 시작하는가
- 단계 4 종료: 사람 간 DM이 제품의 기본 대화 구조가 되었는가
- 단계 5 종료: 클론이 독립 캐릭터가 아니라 사용자 자산으로 재정의되었는가
- 단계 6 종료: 자동응답이 제품 핵심 차별점으로 시연 가능한가
- 단계 7 종료: AI 개입을 사용자가 통제할 수 있는가
- 단계 8 종료: 음성과 부가 기능이 핵심 흐름을 해치지 않고 붙는가

다음 단계 착수 조건:

- 단계 2 착수 전: 새 앱 셸 또는 진입 구조가 준비되어 있어야 한다.
- 단계 3 착수 전: 공개/비공개 계정 정책이 서비스 레벨에서 고정되어 있어야 한다.
- 단계 4 착수 전: 사용자 프로필과 게시물 구조가 최소 동작해야 한다.
- 단계 5 착수 전: 사람 간 DM이 안정적으로 동작해야 한다.
- 단계 6 착수 전: 사용자별 클론/보이스 1개 구조가 고정되어 있어야 한다.
- 단계 7 착수 전: 텍스트 자동응답이 정책대로 저장/표시되어야 한다.
- 단계 8 착수 전: AI 숨김 및 텍스트 음성 재생이 안정적으로 동작해야 한다.

## 7. 기술적 주의사항

- 기존 `ChatConversation`을 억지로 SNS용 DM으로 확장하지 말고, 새 DM 도메인을 설계하는 편이 안전하다.
- 기존 프론트 `CloneStudioPage` 중심 구조를 그대로 키우기보다 `Home / Search / DM / Profile / Settings` 기반 앱 셸 구조로 재편하는 것이 바람직하다.
- 자동응답은 반드시 텍스트 기준으로 먼저 완성하고, 이후 음성 자동응답으로 확장한다.
- 사용자별 마지막 활동 시각은 서버 기준으로 저장하고 판정해야 한다.
- AI 숨김은 개인 상태이므로 메시지 원본 삭제와 혼동되면 안 된다.
- 논쟁 기능은 재사용할 수 있지만 메인 제품 구조와 분리해서 다뤄야 한다.

## 8. 리스크와 대응 전략

### 리스크 1. 기존 코드에 과도하게 끌려갈 위험

- 대응:
  - 재사용 가치가 높은 인증, OpenAI 호출, TTS 계층만 우선 재사용한다.
  - SNS/DM 중심축과 맞지 않는 화면/도메인은 새로 만든다.

### 리스크 2. 자동응답 정책이 뒤늦게 흔들릴 위험

- 대응:
  - `PRD.md`의 정책을 테스트 케이스로 전환한다.
  - 설정값, 부재중 판정, AI 루프 방지 규칙을 먼저 고정한다.

### 리스크 3. 음성 기능이 일정 전체를 지연시킬 위험

- 대응:
  - 음성은 텍스트 자동응답 이후 단계로 분리한다.
  - 텍스트 기반 DM의 완성도를 먼저 확보한다.

### 리스크 4. 기존 기능과 신규 기능의 혼재로 UX가 혼란스러울 위험

- 대응:
  - 초기 단계부터 메인 내비게이션을 SNS 중심으로 재설계한다.
  - 논쟁 같은 재미 기능은 프로필 하위로 이동시킨다.

## 9. 완료 후 산출물 기준

이 계획에 따라 MVP 구현이 완료되면 다음 상태여야 한다.

- 사용자는 SNS 계정을 운영할 수 있다.
- 사용자는 게시물을 올리고 다른 사용자와 관계를 맺을 수 있다.
- 사용자는 사람과 DM을 할 수 있다.
- 사용자는 자신의 클론과 보이스를 설정할 수 있다.

- 사용자는 AI 개입 메시지를 인식하고, 필요 시 숨길 수 있다.
- 사용자는 텍스트와 음성 메시지 형태로 상호작용할 수 있다.
